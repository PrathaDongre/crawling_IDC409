<div class="thumb tright"><div class="thumbinner" style="width:122px;"><a href="/wiki/File:Simple_cycle_graph.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/2/26/Simple_cycle_graph.svg/120px-Simple_cycle_graph.svg.png" decoding="async" width="120" height="110" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/2/26/Simple_cycle_graph.svg/180px-Simple_cycle_graph.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/2/26/Simple_cycle_graph.svg/240px-Simple_cycle_graph.svg.png 2x" data-file-width="300" data-file-height="275" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Simple_cycle_graph.svg" class="internal" title="Enlarge"></a></div>This undirected cyclic graph can be described by the three unordered lists <span class="nowrap">{b, c</span>}, <span class="nowrap">{a, c</span>}, <span class="nowrap">{a, b</span>}.</div></div></div>
<p>In <a href="/wiki/Graph_theory" title="Graph theory">graph theory</a> and <a href="/wiki/Computer_science" title="Computer science">computer science</a>, an <b>adjacency list</b> is a collection of unordered lists used to represent a finite <a href="/wiki/Graph_(discrete_mathematics)" title="Graph (discrete mathematics)">graph</a>. Each unordered list within an adjacency list describes the set of neighbors of a particular <a href="/wiki/Vertex_(graph_theory)" title="Vertex (graph theory)">vertex</a> in the graph. This is one of several commonly used representations of <a href="/wiki/Graph_(abstract_data_type)" title="Graph (abstract data type)">graphs</a> for use in computer programs.
</p>

<h2><span class="mw-headline" id="Implementation_details">Implementation details</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Adjacency_list&amp;action=edit&amp;section=1" title="Edit section: Implementation details">edit</a><span class="mw-editsection-bracket">]</span></span></h2>

<p>An adjacency list representation for a graph associates each vertex in the graph with the collection of its neighbouring vertices or edges. There are many variations of this basic idea, differing in the details of how they implement the association between vertices and collections, in how they implement the collections, in whether they include both vertices and edges or only vertices as first class objects, and in what kinds of objects are used to represent the vertices and edges.
</p>
<ul><li>An implementation suggested by <a href="/wiki/Guido_van_Rossum" title="Guido van Rossum">Guido van Rossum</a> uses a <a href="/wiki/Hash_table" title="Hash table">hash table</a> to associate each vertex in a graph with an <a href="/wiki/Array_data_structure" class="mw-redirect" title="Array data structure">array</a> of adjacent vertices. In this representation, a vertex may be represented by any hashable object. There is no explicit representation of edges as objects.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup></li>
<li>Cormen et al. suggest an implementation in which the vertices are represented by index numbers.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> Their representation uses an array indexed by vertex number, in which the array cell for each vertex points to a <a href="/wiki/Singly_linked_list" class="mw-redirect" title="Singly linked list">singly linked list</a> of the neighboring vertices of that vertex. In this representation, the nodes of the singly linked list may be interpreted as edge objects; however, they do not store the full information about each edge (they only store one of the two endpoints of the edge) and in undirected graphs there will be two different linked list nodes for each edge (one within the lists for each of the two endpoints of the edge).</li>
<li>The <a href="/wiki/Object_oriented" class="mw-redirect" title="Object oriented">object oriented</a> incidence list structure suggested by Goodrich and Tamassia has special classes of vertex objects and edge objects. Each vertex object has an instance variable pointing to a collection object that lists the neighboring edge objects. In turn, each edge object points to the two vertex objects at its endpoints.<sup id="cite_ref-A_3-0" class="reference"><a href="#cite_note-A-3">&#91;3&#93;</a></sup> This version of the adjacency list uses more memory than the version in which adjacent vertices are listed directly, but the existence of explicit edge objects allows it extra flexibility in storing additional information about edges.</li></ul>
<h2><span class="mw-headline" id="Operations">Operations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Adjacency_list&amp;action=edit&amp;section=2" title="Edit section: Operations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The main operation performed by the adjacency list data structure is to report a list of the neighbors of a given vertex. Using any of the implementations detailed above, this can be performed in constant time per neighbor. In other words, the total time to report all of the neighbors of a vertex <i>v</i> is proportional to the <a href="/wiki/Degree_(graph_theory)" title="Degree (graph theory)">degree</a> of <i>v</i>
</p><p>It is also possible, but not as efficient, to use adjacency lists to test whether an edge exists or does not exist between two specified vertices.  In an adjacency list in which the neighbors of each vertex are unsorted, testing for the existence of an edge may be performed in time proportional to the minimum degree of the two given vertices, by using a <a href="/wiki/Sequential_search" class="mw-redirect" title="Sequential search">sequential search</a> through the neighbors of this vertex. If the neighbors are represented as a sorted array, <a href="/wiki/Binary_search" class="mw-redirect" title="Binary search">binary search</a> may be used instead, taking time proportional to the logarithm of the degree.
</p>
<h2><span class="mw-headline" id="Trade-offs">Trade-offs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Adjacency_list&amp;action=edit&amp;section=3" title="Edit section: Trade-offs">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The main alternative to the adjacency list is the <a href="/wiki/Adjacency_matrix" title="Adjacency matrix">adjacency matrix</a>, a <a href="/wiki/Matrix_(mathematics)" title="Matrix (mathematics)">matrix</a> whose rows and columns are indexed by vertices and whose cells contain a Boolean value that indicates whether an edge is present between the vertices corresponding to the row and column of the cell. For a <a href="/wiki/Sparse_graph" class="mw-redirect" title="Sparse graph">sparse graph</a> (one in which most pairs of vertices are not connected by edges) an adjacency list is significantly more space-efficient than an adjacency matrix (stored as a two-dimensional array): the space usage of the adjacency list is proportional to the number of edges and vertices in the graph, while for an adjacency matrix stored in this way the space is proportional to the square of the number of vertices. However, it is possible to store adjacency matrices more space-efficiently, matching the linear space usage of an adjacency list, by using a <a href="/wiki/Hash_table" title="Hash table">hash table</a> indexed by pairs of vertices rather than an array.
</p><p>The other significant difference between adjacency lists and adjacency matrices is in the efficiency of the operations they perform. In an adjacency list, the neighbors of each vertex may be listed efficiently, in time proportional to the degree of the vertex. In an adjacency matrix, this operation takes time proportional to the number of vertices in the graph, which may be significantly higher than the degree. On the other hand, the adjacency matrix allows testing whether two vertices are adjacent to each other in constant time; the adjacency list is slower to support this operation.
</p>
<h2><span class="mw-headline" id="Data_structures">Data structures</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Adjacency_list&amp;action=edit&amp;section=4" title="Edit section: Data structures">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>For use as a data structure, the main alternative to the adjacency list is the adjacency matrix. Because each entry in the adjacency matrix requires only one bit, it can be represented in a very compact way, occupying only <span class="texhtml">&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;<sup>2</sup>/8</span> bytes of contiguous space, where <span class="texhtml">&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;</span> is the number of vertices of the graph. Besides avoiding wasted space, this compactness encourages <a href="/wiki/Locality_of_reference" title="Locality of reference">locality of reference</a>.
</p><p>However, for a sparse graph, adjacency lists require less space, because they do not waste any space to represent edges that are not present. Using a naïve array implementation on a 32-bit computer, an adjacency list for an undirected graph requires about <span class="texhtml">2⋅(32/8)&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124; = 8&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124;</span> bytes of space, where <span class="texhtml">&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124;</span> is the number of edges of the graph.
</p><p>Noting that an undirected simple graph can have at most <span class="texhtml">(&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;<sup>2</sup>&#8722;&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;)/2 ≈ <i>V</i><sup> 2</sup></span> edges, allowing loops, we can let <span class="texhtml"><i>d</i> = &#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124;/&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;<sup>2</sup></span> denote the density of the graph. Then, <span class="texhtml">8&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124; &gt; &#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;<sup>2</sup>/8</span> when <span class="texhtml">&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>E</i></span>&#124;/&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;<sup>2</sup> &gt; 1/64</span>, that is the adjacency list representation occupies more space than the adjacency matrix representation when <span class="texhtml"><i>d</i> &gt; 1/64</span>. Thus a graph must be sparse enough to justify an adjacency list representation.
</p><p>Besides the space trade-off, the different data structures also facilitate different operations. Finding all vertices adjacent to a given vertex in an adjacency list is as simple as reading the list. With an adjacency matrix, an entire row must instead be scanned, which takes <span class="texhtml"><i>O</i>(&#124;<span class="nowrap" style="padding-left:0.1em; padding-right:0.1em;"><i>V</i></span>&#124;)</span> time. Whether there is an edge between two given vertices can be determined at once with an adjacency matrix, while requiring time proportional to the minimum degree of the two vertices with the adjacency list.
</p>
